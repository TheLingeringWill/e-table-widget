## Step 4 — Cancel-from-email-link — PASS (widget side; upstream §9.1c)

Navigate `/1/reservation/112/cancel`.

- Page loads via BFF `/cancel/+page.server.ts`:
  - GET /restaurants/1/bookings/112 → 200 (availableTransitions includes
    "canceled")
  - Aggregate fetched for restaurant.name
- Renders "Réservation du 30/04/2026 à 12:00 pour 2 personne(s).", confirm
  button "Annuler la réservation".
- Click button → BFF action calls setBookingStatus(112, 'canceled') →
  upstream PATCH /restaurants/1/bookings/112/status → **403**
  "operation requires authenticated member" (PRD §9.1c — staff-only).
- UI shows: "Une erreur s'est produite lors de l'annulation." — clean
  error degradation.

The widget side of step 4 is fully wired correctly; only the upstream
allow-list blocks the actual cancel.

## Step 5 — Reconfirm-from-email-link — PASS (degenerate)

Navigate `/1/reservation/112/reconfirm`.

- Page loads via BFF reconfirm/+page.server.ts:
  - GET /restaurants/1/bookings/112 → 200
  - `availableTransitions` for booking 112 does NOT include "reconfirmed"
    (it's `['confirmed', 'canceled', 'refused', 'requires_payment_intent']`),
    so the BFF skips the PATCH and just renders the booking-confirmed shape.
- UI shows: "Merci, votre réservation est bien confirmée. Nous vous
  attendons le 30/04/2026 à 12:00".

Behaviour matches PRD §6.6 ("checks `availableTransitions` to decide
whether to render the action button"). The actual `reconfirmed` transition
can't be exercised end-to-end here because:
1. Booking 112's state machine doesn't allow it.
2. Even if it did, PATCH /status is staff-only on the upstream (§9.1c).
