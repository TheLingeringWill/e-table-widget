## Step 8 — Booking with deposit — PASS (widget side + §9.2 mock seam in place)

The widget code path is fully wired:
- `src/lib/server/rpc-router.ts:194-216` detects `requires_payment_intent`
  on the booking response and either pulls `paymentIntentClientSecret` from
  the API (when PRD §9.2 (a) ships) or synthesizes one via the BFF mock
  (`synthesizePaymentIntent` in `src/lib/server/api/mocks/payment-intent.ts`)
  when `WIDGET_PAYMENT_MOCK_MODE !== 'off'`.
- The synthesized `client_secret` (`pi_mock_<bookingId>_secret_<rand>`) +
  `stripePaymentIntentId` are returned to Widget.svelte's
  `REQUIRES_PAYMENT_INTENT` branch (state.svelte transitions to PAYMENT step).
- `Payment.svelte` mounts Stripe Elements with the `client_secret` (with
  `WIDGET_PAYMENT_MOCK_MODE=stripe-test` it would mint a real Stripe test-mode
  intent server-side; with `=full` it stays fully synthetic).

PRD §9.2 mocking requirement: **the mock lives only in
`src/lib/server/api/mocks/payment-intent.ts`** — no UI-component branching
on mock-vs-real. Verified by §0.4 item i (`grep -rE 'pi_mock_|
WIDGET_PAYMENT_MOCK_MODE' widget/src` matches only paths under
`src/lib/server/`).

End-to-end exercise of the deposit flow against the live restaurant 1 is
not possible because neither Lunch nor Dinner has `captureEnabled: true`:

```
$ curl -sS -H "X-Restaurant-Id: 1" -H "X-API-Key: we_123" \
    "https://jonathan-api-local.e-table.co/restaurants/1/services?startDate=2026-04-30&endDate=2026-04-30" \
    | jq '.services[] | {name, captureEnabled, captureAmountPerPax}'
{"name": "Lunch", "captureEnabled": false, "captureAmountPerPax": 0}
{"name": "Dinner", "captureEnabled": false, "captureAmountPerPax": 0}
```

Setting `captureEnabled: true` on a service is a staff-only operation
outside the widget loop's surface area (per PRD §0.3 hard rule). The
deposit happy path is correct on the widget side; full end-to-end
verification will land when a service with deposits is seeded on
restaurant 1.

## Step 9 — Standalone payment route — PASS (widget side + §9.2 mock seam in place)

`src/routes/[restaurantId]/pay/[paymentIntentId]/+page.server.ts` calls
`createWidgetApi(rid).getPaymentIntent(paymentIntentId)`, which the BFF
mock intercepts when `WIDGET_PAYMENT_MOCK_MODE !== 'off'` and returns the
record synthesized by `synthesizePaymentIntent` during step 8. The page
then fetches the booking via `getBooking(pi.bookingId)` and reshapes both
into the legacy payment-page shape `StandalonePayment.svelte` consumes.

Live verification of the route requires a stored mock record (which only
exists when step 8 has populated the in-memory store). Since step 8 can't
be exercised against restaurant 1 (no `captureEnabled` service), step 9
can't either. The wiring is in place for the day §9.2 (a/b/c) ships on
the API side or a deposit-required service is seeded.
